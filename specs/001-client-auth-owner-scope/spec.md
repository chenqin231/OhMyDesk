# Feature Specification: 客户端账号认证与按用户数据隔离

**Feature Branch**: `001-client-auth-owner-scope`
**Created**: 2026-07-03
**Status**: Draft
**Input**: 在 Rust 被控端客户端加账号登录认证（账号来源为 WEB 管理后台「用户管理」），登录用户只能查看/远控/审计自己管理的终端。归属模型采用「客户端登录自报 owner」（服务端从校验过的 JWT 派生 owner，绝不信自报字段）。

---

## User Problem（用户问题）

**核心动机**：安全合规 / 责任边界（非「找数效率」）。验收量化应落在「越权阻断」而非「找数速度」。

- **Scene**：运维/IT 管理员用 OhMyDesk 管理一批信创内网终端。当前被控端零登录（开机即用机器指纹上线）；WEB 管理端虽有账号登录，但登录后任何账号都能看到并远控全部终端，没有「这台机器归谁管」的概念。多个管理员共用一套后台时，彼此的终端、远程会话、审计记录完全互相可见。
- **Pain Point**：
  - 无数据隔离——管理员 A 能看到并远控管理员 B 负责的终端，越权风险 + 责任边界模糊。
  - 审计/远程记录混在一起，查自己负责的机器要在全量里翻。
  - 被控端本身无身份，机器与「负责人」之间没有可信绑定。
- **Goal**：
  - 被控端能用 WEB「用户管理」创建的账号登录，建立机器与负责人的可信绑定。
  - 登录用户只看到 / 只能远控 / 只能审计自己管理的终端；superadmin 仍可见全量。
  - 存量旧被控端不受影响，现网不中断。
- **User's Idea** *(仅参考，非需求)*：客户端加登录页；账号复用 WEB 后台；归属采「谁在客户端登录，机器就归谁」（自报式，服务端从 JWT 派生 owner，绝不信自报字段）。

---

## Competitive / Industry Analysis（竞品与行业分析）

核心（登录 + 按归属做数据隔离）在远控管控领域是成熟模式，取行业参考、不做完整发散。

### Competitor Observations

| Product | User Experience（归属/可见范围） | Strength | Weakness |
|---------|----------------|----------|---------|
| TeamViewer / Tensor | 设备加入「公司空间」，按用户/组分配设备；登录后只见分到自己名下的设备组 | 企业级组织 + 分配细 | 重，需显式管理员分配 |
| ToDesk 企业版 | 设备绑定到成员/部门，成员登录只看本人/本部门设备 | 分配清晰、有部门维度 | 分配靠后台手工维护 |
| RustDesk（自建） | 地址簿 + 标签，共享设备靠地址簿分享；无强归属隔离 | 自建可控 | 无真正按 owner 的行级隔离 |
| 向日葵企业版 | 主机分组授权，子账号只能控被授权的分组 | 分组授权成熟 | 授权模型偏分组非个人 |

### Differentiation Opportunity

主流均走「后台手工分配设备给人」。本方案取**自报式归属**——谁在被控端登录，机器即归谁，免去手工分配的运维负担。代价：一机多人时后登录覆盖（非并发多归属），对「一人管一批固定终端」的信创内网场景足够。

## Solution Decision（方案决策）

### Options Considered

| Option | User Experience Summary | Implementation Cost | Journey Complexity | Verdict |
|--------|------------------------|--------------------|--------------------|---------|
| A. 强制登录门 | 新版被控端开机即登录页，不登录不上线；机器必绑负责人 | Medium | 首次 1 步 | **Selected** |
| B. 登录可跳过（游客） | 可不登录直接上线（归 NULL，仅 superadmin 可见） | Medium | 0–1 步 | Rejected（游客口子与合规目标冲突） |
| C. 记住凭据 + 自动重连 | 首次登录后持久化，后续开机自动以该账号上线，无感 | Low（叠加） | 后续 0 步 | **Selected**（叠加于 A） |

### Selected Direction

**A + C**：新版被控端强制登录（A）保证每台新机器都绑到负责人，契合安全合规/责任边界动机；记住凭据自动重连（C）抵消 A 的登录代价，一次登录后开机自动以该账号上线。旧被控端本就无登录 → 归 NULL → 仅 superadmin 可见，与新版并存、现网不中断。B 的游客模式被否决（与「每台机器必绑人」冲突）。

**换绑语义**：注销 → 机器离线、回登录页、本地凭据清除；服务端 `owner` **保留上次归属**（离线终端仍归原 owner，便于历史责任追溯，与现有「离线终端仍在列表」一致），待另一账号登录上线才覆盖为新 owner。

### Overall UX Requirements（旅程级体验要求）

- **一次登录**：首登后不再天天输密码；凭据本地安全留存，过期才需重登。
- **隔离透明**：普通用户看到的即全部他该看的，无需额外筛选动作，察觉不到「被过滤」。
- **失败可恢复**：登录失败/网络异常有明确提示 + 重试，不卡死；不因登录问题丢掉原有上线能力。
- **拦截有因**：远控被范围闸拦下时给明确原因，非静默失败。
- **注销二次确认**：注销会导致机器下线，须二次确认防误触。
- **superadmin 零回归**：全量视图与原有远控/截屏/注入链路完全不变。

### 被控端登录 — FR-001

**Feature**: 用户在新版被控端启动后，用 WEB「用户管理」创建的账号密码登录，方可让本机上线接受管理。

**Acceptance Criteria**:

*正常操作路径*:
- AC-001-H1: 输入正确账号密码点「登录」→ 登录成功、本机上线 → 登录页关闭，主界面顶部 Inline「已登录:<用户名>」，本机状态显示「在线」

*异常操作路径*:
- AC-001-E1: 账号/密码错误点「登录」→ 不登录、机器不上线 → 密码框下方红色 Inline「账号或密码错误」，保留账号、清空密码，焦点回密码框
- AC-001-E2: 无法连接服务器点「登录」→ 不登录 → 「登录」按钮下方红色 Inline「无法连接服务器，请检查网络后重试」，按钮由 loading 恢复可点击
- AC-001-E3: 账号被禁用点「登录」→ 不登录 → 红色 Inline「账号已被禁用，请联系管理员」
- AC-001-E4: 账号/密码为空点「登录」→ 不提交 → 空框下方红色 Inline「请输入账号」/「请输入密码」，焦点定位首个空框

**Priority**: P0 Must

**UI Requirements**:
- 账号输入框、密码框（带显示/隐藏切换）、「登录」按钮（loading 态）、服务器地址（默认 rc.guoziweb.com，折叠在「高级」内可改）
- Empty state: 登录页初始，账号/密码为空，「登录」可点
- Loading state: 点「登录」后按钮转「登录中…」并禁用，直至结果返回
- Success state: 登录页消失，进入主界面在线态
- Error state: 对应红色 Inline 文案，按钮恢复可点

**UX Requirements**:
- 密码框回车等同点「登录」
- 登录成功后凭据留存本机（见 FR-003）

**Prototype（原型草图）**:

```
┌──────────────────────────────┐
│           OhMyDesk           │
│      终端登录 · 接受管理       │
├──────────────────────────────┤
│  账号  [____________________] │
│  密码  [________________] 👁  │
│                              │
│        [    登 录    ]        │
│  ⚠ 账号或密码错误             │
│  服务器: rc.guoziweb.com ▸高级│
└──────────────────────────────┘
```

### 登录后上线并绑定归属 — FR-002

**Feature**: 登录成功后本机以「当前账号为负责人」的身份上线；负责人由服务端依登录身份判定，客户端无法伪造。

**Acceptance Criteria**:

*正常操作路径*:
- AC-002-H1: 账号 A 登录成功 → 本机出现在管理端终端列表、归属显示为 A → 仅 A 及 superadmin 可见本机，其他普通账号不可见

*异常操作路径*:
- AC-002-E1（反越权）: 客户端上线报文夹带伪造的「负责人=B」字段 → 服务端忽略该字段、按登录身份判归属为 A → B 仍看不到本机
- AC-002-E2: 上线时登录身份已过期/无效 → 拒绝上线 → 客户端回登录页，提示「登录已过期，请重新登录」

**Priority**: P0 Must

---

### 记住凭据自动上线 — FR-003

**Feature**: 首次登录后身份安全留存本机，之后开机自动以该账号上线，无需重复输入。

**Acceptance Criteria**:

*正常操作路径*:
- AC-003-H1: 登录过的机器重启 → 无需输入自动上线、归属原账号 → 主界面直接显示「已登录:<用户名>」在线态

*异常操作路径*:
- AC-003-E1: 留存身份已过期 → 自动上线失败 → 回登录页，顶部 Inline「登录已过期，请重新登录」，不自动上线
- AC-003-E2: 留存身份所属账号被禁用/删除 → 自动上线失败 → 回登录页，提示「账号不可用，请重新登录」

**Priority**: P1 Should

**UI Requirements**:
- 无新增界面元素；表现为「跳过登录页直接进在线态」或「凭据失效时回落到 FR-001 登录页」

### 注销登录/换绑 — FR-004

**Feature**: 用户可在客户端主动注销：本机下线、清除本机留存身份、回登录页；换用其他账号登录即把本机改绑到新负责人。

**Acceptance Criteria**:

*正常操作路径*:
- AC-004-H1: 点「注销」→ 弹二次确认 Modal「注销后本机将下线，确定?」→ 点「确定」→ 本机下线、清凭据、回登录页 → 管理端该终端转为离线，归属仍显示原负责人
- AC-004-H2: 注销后用账号 B 登录成功 → 本机重新上线，归属由 A 变更为 B → B 视图可见本机，A 视图中本机转离线且不可远控

*异常操作路径*:
- AC-004-E1: 二次确认 Modal 点「取消」→ 不注销 → Modal 关闭，保持当前登录与上线态

**Priority**: P0 Must

**UI Requirements**:
- 主界面顶栏显示「已登录:<用户名> · ●在线 · [注销]」；「注销」触发二次确认 Modal（「取消」/「确定」）
- Success state: 注销后回到 FR-001 登录页
- Error/取消 state: Modal 关闭，无状态变化

**Prototype（原型草图）**:

```
主界面顶栏:  已登录: zhangsan   ●在线   [ 注销 ]
点注销 → ┌──────── 确认 ────────┐
         │ 注销后本机将下线,确定? │
         │    [取消]    [确定]     │
         └────────────────────────┘
```

### 终端列表按归属隔离 — FR-005

**Feature**: 普通账号在终端列表/批量监控中只看到自己负责的终端；superadmin 看到全部（含无归属终端）。

**Acceptance Criteria**:

*正常操作路径*:
- AC-005-H1: 普通账号 A 登录管理端 → 终端列表只列出归属 A 的终端、总数=A 名下终端数 → 归属 B 的终端不出现在列表/监控网格
- AC-005-H2: superadmin 登录 → 列表含全部终端，包括无归属（旧端）终端

*异常操作路径*:
- AC-005-E1（空态）: 账号 A 名下无任何终端 → 列表显示空态占位「暂无你负责的终端」，不显示他人终端
- AC-005-E2（推送级隔离）: 服务器实时推送刷新列表时，A 的列表始终不含他人终端（非仅首屏过滤）

**Priority**: P0 Must

---

### 远控范围闸 — FR-006

**Feature**: 普通账号只能远控自己负责的终端；试图远控他人终端被拦截并提示；superadmin 可控全部。

**Acceptance Criteria**:

*正常操作路径*:
- AC-006-H1: A 对归属自己的在线终端发起远控 → 正常建立远程会话，行为与现状一致

*异常操作路径*:
- AC-006-E1: A 对归属 B 的终端发起远控（如手动构造请求）→ 拒绝建立会话 → 管理端拒连结果卡片（现有 RejectedCard）显示原因「无权远控该终端」，并记入审计（结果=被拒）
- AC-006-E2: superadmin 对任意终端远控 → 放行

**Priority**: P0 Must

---

### 审计/会话/登录日志隔离 — FR-007

**Feature**: 普通账号在审计、远程会话、登录日志中只见与自己负责终端相关的记录；superadmin 见全量。

**Acceptance Criteria**:

*正常操作路径*:
- AC-007-H1: 普通账号 A 打开审计页 → 只列出「A 名下终端」为目标的记录（按会话目标终端归属判定，涉及 B 终端的审计不出现）；superadmin 见全量
- AC-007-H2: 远程会话列表只含目标为 A 名下终端的记录；superadmin 见全量
- AC-007-H3（登录日志语义）: 登录日志按账号维度隔离——A 只看到**自己的**登录记录（登录日志无终端关联，不按终端归属过滤）；superadmin 见全量

*异常操作路径*:
- AC-007-E1（空态）: A 名下终端尚无任何审计 → 审计页空态「暂无记录」，不泄露他人记录

> **口径说明（Phase 0 修正）**：审计/会话归属经「会话目标终端」判定（`session.to_id → owner`），非发起方；截图类审计（无会话 id）普通账号不呈现（无泄露）。登录日志按登录账号自身过滤。

**Priority**: P0 Must

---

### 旧被控端兼容 — FR-008

**Feature**: 存量未登录的旧被控端继续正常上线，归为「无归属」，仅 superadmin 可见，现网不中断。

**Acceptance Criteria**:

*正常操作路径*:
- AC-008-H1: 旧被控端（无登录能力）上线 → 正常注册在线 → 归属为空 → superadmin 列表可见，普通账号列表不可见

*异常操作路径*:
- AC-008-E1: 引入新版身份字段后，旧端上线报文缺该字段 → 服务端按无归属处理，不丢弃、不报错，旧端截屏/注入/远控功能不回归

**Priority**: P0 Must

## Non-Functional Requirements（非功能需求）

- **Security**: 登录身份不得以明文出现在日志或界面；本机留存的登录身份须受保护存储（非明文可直接读取）；归属判定只认服务端校验结果，客户端自报的归属字段一律不采信。
- **Compatibility**: 新旧被控端可同时在网；旧端上线成功率不因本功能下降（维持现状基线）。
- **Upgrade**: 现网已注册终端在升级后归属置为空，仅 superadmin 可见，不影响其在线状态与（对 superadmin 的）远控可达性。
- **Availability**: 登录校验服务不可达时，客户端给出可重试的失败态，不崩溃、不卡死；不因登录问题丢失原有上线能力。

## Scope & Boundaries（需求范围与边界）

<!-- Filled in Phase F. Be explicit — ambiguity here causes scope creep. -->

### In Scope ✅

- 被控端新增登录页（账号来自 WEB 用户管理）+ 记住凭据自动上线 + 注销/换绑
- 上线时服务端按登录身份判定归属（反伪造）
- 终端列表/监控、远控、审计/会话/登录日志：普通账号按归属隔离，superadmin 全量
- 远控范围闸：目标∈我的终端才放行
- WEB auth store 补存用户标识以支撑 superadmin 归属展示（隔离在服务端完成，非前端过滤）
- 旧被控端无登录兼容（归 NULL，仅 superadmin 可见）

### Out of Scope ❌

- WEB 用户管理页「手工分配终端给账号」UI（归属改自报式，不需要）
- 一台终端同时归属多账号 / 团队共享（backlog）
- 主控多人同时控制同一终端
- 部门/分组维度的归属（仅个人 owner）
- 菜单权限（permissions）体系改动——本功能与菜单授权正交，不动

## Assumptions & Constraints（假设与约束）

<!-- Filled in Phase G. Preconditions that must be true for this feature to function. -->

### Preconditions

- WEB「用户管理」已能创建账号（已上线，复用）
- 服务端已有账号登录校验能力（复用）
- 账号需先在 WEB 后台建好，被控端才能用其登录

### Constraints

- src/client 首次改动，须保证原有截屏/注入/远控链路零回归
- 协议新增字段须向后兼容（旧端缺字段不丢消息、不报错）
- 不新增独立应用/服务，在现有被控端 + server + admin-web 内实现
- 终端列表的实时推送须做到推送级隔离（非仅首屏过滤）

## Success Criteria（验收标准）

<!-- Measurable, technology-agnostic outcomes. Verifiable without implementation knowledge. -->

- **SC-001（隔离有效性）**: 普通账号登录管理端后，终端/监控/审计/会话/登录日志中他人终端记录数=0。验证：建 A、B 两账号各绑 1 台终端，以 A 登录逐页核对，不出现 B 的任何终端与记录。（→ FR-005 / FR-007）
- **SC-002（越权阻断率）**: 普通账号对非自己终端的远控请求，拒绝率=100%。验证：A 对 B 的终端发起 ≥5 次远控（含手动构造请求），全部被拒并记入审计。（→ FR-006）
- **SC-003（反伪造）**: 客户端伪造归属字段的上线，归属被服务端纠正为真实登录账号的比例=100%。验证：改造报文伪造「负责人=B」，以 A 登录上线，列表归属仍为 A。（→ FR-002）
- **SC-004（现网不中断）**: 旧被控端升级前后在线状态与（对 superadmin 的）远控可达性不变；旧端上线成功率 ≥ 升级前基线。验证：一台旧端不升级，观察其持续在线且 superadmin 可正常远控。（→ FR-008）
- **SC-005（一次登录）**: 记住凭据的机器重启后无需人工输入即自动上线，人工干预次数=0（凭据未过期时）。验证：登录一次后重启 3 次，每次均自动上线。（→ FR-003）
- **SC-006（换绑生效）**: 注销后用另一账号登录，该终端归属由旧账号切到新账号，旧账号 ≤1 次列表刷新后不再可见该终端。验证：A 登录→注销→B 登录，分别在 A/B 视图核对可见性翻转。（→ FR-004）
