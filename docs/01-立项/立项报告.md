## 一、立项背景与问题

### 1.1 市场真实痛点

企业远程控制市场按**网络边界**裂为两半，中间存在明确的市场空白：

| 品类 | 代表产品 | 能力 | 致命短板 |
|------|---------|------|---------|
| 消费级远控 | RustDesk(117k★)、TeamViewer、向日葵 | 体验好、跨平台 | 依赖公网中转，物理隔离内网**无法使用**；无 B 端审计 |
| 服务器特权访问 | 堡垒机、Teleport(20.5k★) | 审计强 | 仅管服务器，**触及不到员工终端 PC** |
| **空白区** | — | — | **内网员工终端 PC 的远程协助 + 安全管控 + 合规审计** |

### 1.2 信创国产化叠加结构性需求

党政「2026 年底 80% 国产化」要求：麒麟/统信 UOS + 龙芯/鲲鹏/飞腾架构平台上，主流远控软件适配严重缺位，信创终端管控问题日益突出。

### 1.3 最接近竞品分析

GitHub 上最接近的开源项目 **MeshCentral**（6.8k★）：Agent 反连 + 自托管 Web + 设备分组 + 审计，架构范式被本项目借鉴；但无信创适配、无 AI 能力、Star 量远不及主流远控。

---

## 二、项目目标

**填补内网物理隔离场景下，信创终端远控 + 管控 + 审计一体化能力的市场空白。**

核心差异化（每条对应真实痛点）：

1. **数据不出网**：Agent 反连自托管，纯内网部署，物理隔离网直接可用
2. **信创原生**：终端自动标注麒麟/统信/龙芯/鲲鹏等信创 OS 和 CPU，面向国产化评估
3. **管控审计闭环**：连接/操作纯文本审计，堡垒机管不到的员工终端 PC 纳入管控
4. **AI 时代数据消费**：内置 MCP Server，管控数据供内网 AI 自然语言查询

## 三、产品功能范围（5 大模块）

| 模块 | 功能概述 |
|------|---------|
| **M1 终端资产管理** | Agent 反连注册，采集硬件（CPU/内存/GPU/IP/MAC），自动识别信创标识，在线/离线监控 |
| **M2 远程控制** | Web 管理端→终端（模式 A）/ 终端→终端（模式 B），画面流 + 键鼠注入 + 授权弹窗 |
| **M3 批量监控截图** | 一键下发全网截图指令，所有在线终端截图墙（核心视觉记忆点） |
| **M4 会话审计** | 纯文本连接/操作记录，落 SQLite，按终端/时间筛选 |
| **M5 MCP/AI 问答** | 内置 MCP Server，5 只读工具，自然语言查全网终端态势 |

## 四、技术方案

### 4.1 技术栈选型

| 组件 | 技术 | 选型依据 |
|------|------|---------|
| Agent 客户端 | Rust + Slint（软渲染） | 无 GPU 依赖，绕开国产 CPU 图形栈；xcap 截屏 + enigo 注入 |
| 服务端 Relay | Rust + axum + tokio + tokio-tungstenite | 与客户端同栈；rustls 纯 Rust TLS，避免 openssl 交叉编译坑 |
| 共享协议 | Rust serde + ts-rs | 单一事实源，三端类型零漂移 |
| 管理端 Web | React + Vite + Tailwind + shadcn/ui | 浏览器限制必须 TS；Claude 最熟，死线最稳 |
| MCP Server | TypeScript + @modelcontextprotocol/sdk | TS SDK 最成熟，薄层设计 |
| 数据库 | SQLite（ohmydesk.db，单文件，零外部依赖） | demo 阶段最低配置门槛 |
| 信创扩展 | loongarch64/aarch64 交叉编译（rustls） | 参考 RustDesk 麒麟/统信落地实证 |

### 4.2 架构概要

```
┌────────────┐  WS(反连)  ┌──────────────────────────┐  HTTP/stdio  ┌────────────┐
│ Agent 客户端 │ ─────────▶ │      Server (Relay)       │ ◀──────────  │ MCP Server │
│ Slint+Rust  │ ◀───────── │ 注册表/鉴权/路由/审计/静态 │              │   TS 薄层   │
└────────────┘  帧/键鼠   └──────────┬───────────────┘              └────────────┘
                                     │ HTTP+WS+UI
                               ┌─────▼──────┐
                               │  管理 Web   │ React + Vite
                               └────────────┘
全链路内网，数据不出网
```

## 五、可行性分析

| 维度 | 结论 | 依据 |
|------|------|------|
| 技术可行性 | ✅ | RustDesk 已在信创生产落地；xcap/enigo/sysinfo 跨平台成熟 |
